# Бенчмарки

Воспроизводимое сравнение `dskripchenko/php-pdf` с четырьмя наиболее
используемыми PHP-библиотеками для генерации PDF на Packagist:
`mpdf/mpdf`, `tecnickcom/tcpdf`, `dompdf/dompdf` и `setasign/fpdf`.

## Методология

Каждая пара (библиотека, сценарий) запускается **5 раз** в собственном
PHP-подпроцессе после **1 прогревочного** запуска. Сообщаются медиана
wall-time и пиковая память. Размер выхода — это число байт в
произведённом PDF.

Изоляция в подпроцессе важна: один долго работающий PHP-процесс
накапливает загруженный код и давление OPcache, что раздувает пик
памяти того сценария, который выполнился последним. Старт свежего
процесса на каждый запуск делает колонку памяти осмысленной.

Wall-time замеряется через `hrtime(true)`; память — через
`memory_get_peak_usage(true)`.

**Тестовое окружение:**
- macOS 25.4 (Darwin), Apple Silicon.
- PHP 8.4 CLI (Zend OPcache включён, JIT по умолчанию).
- Версии библиотек (Packagist):
  - dskripchenko/php-pdf 1.0.0
  - mpdf/mpdf ^8.2
  - tecnickcom/tcpdf ^6.7
  - dompdf/dompdf ^3.0
  - setasign/fpdf ^1.8

## Сценарии

| Ключ      | Описание                                               |
|-----------|--------------------------------------------------------|
| `hello`   | Одна страница A4, один абзац `Hello, world!`.         |
| `invoice` | 100 страниц × 50 строк; на каждой странице заголовок `<h1>` и 4-колоночная таблица. |
| `images`  | 20 страниц × 4 JPEG-миниатюры (250 × 180 pt каждая).  |
| `html`    | Один `<h1>`, два `<h2>`-раздела, 24 абзаца Lorem с bold/italic/link плюс маркированные списки. ~5 страниц. |

У FPDF нет поддержки HTML, поэтому в сценарии `html` он пропускается.

## Как воспроизвести

```bash
git clone https://github.com/dskripchenko/php-pdf.git
cd php-pdf

# Install the library under test.
composer install

# Install the comparison harness with mpdf, tcpdf, dompdf, FPDF as
# isolated dev dependencies (does not pollute the library's composer.json).
cd scripts/bench
composer install

# Run all scenarios; ~30 seconds total wall time.
php run.php             # JSON to stdout
php run.php --md        # Markdown table form
```

Лимиты памяти внутри `run.php` / `one.php` зафиксированы на 1 GB, чтобы
дать dompdf запас на сценарии со счётом.

## Замечания и оговорки

- **Преимущества FPDF.** FPDF выигрывает на `hello` и `images`,
  потому что это минимальный императивный API — никакого HTML-парсинга,
  никаких переносов текста, никакого табличного потока, никакого
  UTF-8. Он честен в сравнении только с равно минимальными сценариями;
  сопоставимое покрытие возможностей (вход HTML, графики, штрихкоды,
  шифрование, подпись, PDF/A, AcroForms) в FPDF недоступно.

- **Память dompdf на счёте.** dompdf строит полный DOM и
  CSS-box-дерево в памяти до раскладки, что масштабируется с общим
  размером HTML, а не на страницу. Пик в 350 MB на 100-страничном
  счёте отражает этот архитектурный выбор.

- **Размер выхода.** Меньше — не всегда лучше: одни библиотеки
  встраивают более агрессивные сабсеты шрифтов, другие используют
  несжатые потоки. Все сценарии выше используют дефолтные настройки
  шрифтов/сжатия каждой библиотеки.

- **Покрытие HTML.** В HTML-сценарии используются элементы (`<h1>`,
  `<h2>`, `<p>`, `<strong>`, `<em>`, `<a>`, `<ul>`, `<li>`), которые
  поддерживают все библиотеки. Сложный CSS (Flexbox, многоколоночник,
  обтекания) не замеряется.

- **Накладные расходы первого запуска.** Wall-time включает старт
  подпроцесса и бутстрап автозагрузчика, что реалистично для
  serverless / per-request нагрузок. Долго работающие воркеры увидят
  пропорционально меньшие накладные расходы на отрисовку у всех
  библиотек.
